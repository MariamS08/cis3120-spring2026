Module 15 — Press Release to Plot

Instructor: Patrick Slattery Estimated reading time: 25 minutes

What this module is about

Public companies in the United States are required to disclose material events to the Securities and Exchange Commission. Many of these disclosures concern physical operations: a new distribution center coming online, a chain of retail stores closing, a manufacturing plant relocating to a different state. The information is public, regularly updated, and structured at the level of the filing — but the substance of each announcement is buried inside paragraphs of prose written by corporate counsel. Extracting that substance at scale requires reading the text.

This module presents the pipeline that performs that reading. The pipeline accepts a date range as input and produces an interactive geographic map as output. Between the input and the output, five stages coordinate three external services and one client-side library. Stage 1 retrieves candidate filings from SEC EDGAR. Stage 2 downloads the underlying press release text. Stage 3 sends each press release to the Anthropic API and asks the model to determine whether the filing announces a location event and, if so, to extract the city, state, and event type. Stage 4 looks up latitude and longitude coordinates for each location. Stage 5 renders the result on a folium map.

The module extends the work begun in Module 14 in three concrete directions. It moves from free-form text generation to structured output via prompt engineering. It composes multiple APIs in sequence rather than calling a single one. And it uses authoritative public data — the SEC's own records — rather than synthetic examples.

Why this matters

The skill being practiced here is general. Many business questions take the form: What is happening, where, and at what scale? The answers exist in public text — in regulatory filings, in earnings calls, in trade press, in court records — but the text is unstructured. A human reading every 8-K filing for a single quarter is not a feasible workflow. A pipeline that reads them and returns a structured table is.

The retail and logistics sectors illustrate the point. A regional analyst tracking competitor expansion needs to know which retailers are adding stores in which markets. A supply chain consultant needs to know where new distribution capacity is coming online. A commercial real estate firm needs early warning of which markets are seeing facility closures. None of these questions can be answered from a stock ticker or a press release headline alone; they require reading the body of the filing and extracting the substantive facts.

The pipeline in this module is one instance of a broader pattern: retrieve relevant documents from a corpus, then use a language model to read each one and return structured data. That pattern appears in regulatory monitoring, contract review, customer support triage, and academic research. Module 15 establishes the pattern using a public dataset that students can run themselves at low cost. Module 16 will extend the same pattern to private document corpora using retrieval-augmented generation.

Background concepts

The SEC EDGAR system

EDGAR — the Electronic Data Gathering, Analysis, and Retrieval system — is the SEC's central repository of corporate filings. Every public company in the United States files documents through EDGAR, and every document that EDGAR receives is made publicly available. The full-text search interface accepts a query phrase, a date range, and a form type, and returns the filings that match.

The form type used in this module is Form 8-K, the "current report." Companies file an 8-K whenever a material event occurs between scheduled reports. Item codes inside the form indicate the nature of the event: Item 5.02 covers executive changes, Item 2.01 covers acquisitions, and so on. There is no dedicated item code for facility openings or closings; most such announcements are filed under Item 8.01, which is a general-purpose "Other Events" category. This is the structural reason why keyword search — rather than item-code filtering — is necessary for the question this module asks.

Each 8-K filing is identified by an accession number, a string of digits assigned at the moment of submission. The narrative announcement is usually attached as an exhibit, conventionally Exhibit 99.1. The exhibit is a separate file inside the filing's directory on EDGAR's archive server. Constructing the URL of an exhibit requires the company's CIK (Central Index Key, a numeric company identifier), the accession number with hyphens removed, and the exhibit's filename — all three of which are returned by the search API.

Header requirement. Both the SEC and OpenStreetMap require a descriptive User-Agent header containing a real name and a real email address on every request. Generic agents are rejected with HTTP 403. Students will replace a placeholder with their Baruch email before running the notebook.

Why combined-OR queries fail

The natural way to retrieve filings that match any of several phrases is a single query joining the phrases with the boolean operator OR. This pattern is documented as supported by EDGAR's full-text search engine. In practice, however, a query of the form ("new store" OR "new facility" OR "distribution center") returns zero hits. The behavior is silent: no error is raised, the response is well-formed JSON, and the only signal that something is wrong is the empty result set.

The pipeline therefore queries each phrase independently and merges the results, deduplicating by accession number. This is slower and more verbose than the documented approach would be, but it is the approach that actually works. Discovering such gaps between documented and actual behavior is itself a transferable skill: validate that an external service does what its documentation says before relying on the documentation in production code.

Asking the model for structured output

The language model in Stage 3 is asked to return a JSON object with a fixed schema: a boolean is_location_event, an event_type, a city, a state, and a one-sentence summary. This pattern — sometimes called structured output via prompt engineering or JSON-mode prompting — is the mechanism by which a language model becomes a reliable component of a larger pipeline. Downstream code does not parse free-form prose; it parses a dictionary.

Two design choices in the system prompt deserve attention. First, the prompt instructs the model to return is_location_event: false for filings that mention a location only in passing — for example, the company's headquarters address in the boilerplate at the bottom of every press release. This single instruction is what raises the pipeline's precision from approximately 60 percent to 100 percent in prototype testing. Second, the prompt is explicit about output format: "Return only the JSON object with no preamble, no markdown fences, no explanation." The model is generally compliant with this instruction, but the parsing code in the pipeline is nonetheless defensive — it strips markdown fences if present and falls back to a default record on parse failure.

The model used in this stage is the current Haiku model in the Claude 4.5 family (model identifier claude-haiku-4-5-20251001). Haiku is the appropriate choice because the task is binary classification followed by template-shaped extraction. There is no extended reasoning to perform and no creative writing to produce. A larger model in the same family would cost roughly six times as much, run several times more slowly, and — as the prototype data show — produce slightly lower recall for this particular task.

Geocoding and rate limits

The model returns a city and a state. To plot a marker on a map, a latitude and a longitude are required. The translation from a human-readable place name to geographic coordinates is called geocoding. This pipeline uses Nominatim, the geocoding service operated by the OpenStreetMap project, because it is free, well-documented, and adequate for city-level precision.

Nominatim's terms of use require a maximum of one request per second from any single client. The pipeline enforces this with a 1.1-second pause between calls, deliberately leaving a small margin. Failing to respect this limit results in IP-level rate limiting that is difficult to diagnose, because the affected requests succeed individually but fail in aggregate. The SEC's EDGAR service imposes a similar but more permissive limit of ten requests per second; the pipeline uses a 0.15-second pause between EDGAR calls.

Each stage produces input to the next. The boundaries between stages are deliberate: each one can be tested in isolation, swapped for a different implementation, or rerun without rerunning the upstream stages. Such clean staging is the basic discipline of any production data pipeline.

The recall vs precision tradeoff

Stage 1 is tuned for high recall. The keyword phrases are chosen to capture as many candidate filings as possible, even at the cost of returning many filings that turn out to be irrelevant. Stage 3 is tuned for high precision. The system prompt instructs the model to filter aggressively, returning is_location_event: false for any filing that does not unambiguously announce a location event. This division of labor — broad retrieval followed by strict filtering — is the recall vs precision tradeoff that the pipeline is designed to manage. It is the same pattern that underlies retrieval-augmented generation, which Module 16 will introduce.

What you will produce

The deliverable is a folium HTML map embedded in a Jupyter notebook cell. The map displays one marker per location event identified by the pipeline, with markers color-coded by event type — green for openings, red for closings, orange for relocations, blue for expansions, and gray for events that did not fit into the four primary categories. Each marker carries a popup containing the company name, the event summary written by the model, and a link to the original filing on EDGAR.

A typical run with the configured 30-day window returns approximately 100 candidate filings, of which 15 to 20 are confirmed location events. The end-to-end runtime is approximately four minutes, dominated by the geocoding stage's mandatory one-second pauses. The total cost in API charges is approximately $0.30 per run.

Beyond the map itself, the notebook produces two intermediate artifacts that are useful in their own right: a deduplicated table of every candidate filing with its company, date, and exhibit URL; and a structured table of every confirmed location event with its city, state, and coordinates. Either table can be exported to a spreadsheet for further analysis.

What you should be able to do afterward

After completing the reading and the in-class walkthrough, you will be able to:

  1. Query the SEC EDGAR full-text search API to retrieve filings matching a date range and a set of keywords.
  2. Construct multi-phrase queries that work around the documented but non-functional combined-OR query syntax.
  3. Fetch and clean HTML content from EDGAR exhibits using requests and BeautifulSoup.
  4. Design a system prompt that elicits structured JSON output from Claude.
  5. Parse model output defensively, handling occasional formatting variations.
  6. Geocode US city/state pairs using OpenStreetMap Nominatim with appropriate rate limiting.
  7. Construct an interactive folium map with color-encoded markers and informative popups.
  8. Reason about model selection in terms of task fit, choosing Haiku for classification and extraction work and reserving Opus for tasks that require deep reasoning.
  9. Articulate the recall vs precision tradeoff and explain how a multi-stage pipeline manages it.

Glossary

The pipeline
The five-stage workflow that takes a date range as input and produces an interactive map as output. Used throughout this module to refer to the overall system.
Stage 1 through Stage 5
The five components of the pipeline, in order: EDGAR retrieval, exhibit fetch, Anthropic API classification and extraction, geocoding, and folium rendering. The numbering is fixed; each stage is referred to consistently as "Stage N" in capitalized form.
8-K filing
A "current report" submitted by a public company to the SEC to disclose a material event that occurred between scheduled periodic reports.
Accession number
The unique identifier assigned by EDGAR to each filing at the moment of submission. Used in this module to deduplicate the multi-phrase search results and to construct exhibit URLs. Preferred over the term "filing ID."
CIK (Central Index Key)
A numeric identifier assigned by the SEC to each filer. Used together with the accession number to locate a filing's directory on EDGAR's archive server.
Exhibit 99.1 (or "the exhibit")
The conventional attachment slot used for a press release filed alongside an 8-K. The narrative content of the announcement lives in the exhibit, not in the form itself.
Structured output via prompt engineering (JSON-mode prompting)
The technique of writing a system prompt that constrains a language model to return a JSON object with a fixed schema, so that downstream code can parse the output as a dictionary rather than as free-form prose.
The recall vs precision tradeoff
The fundamental tension in any retrieval system between returning too many irrelevant items (low precision) and missing relevant items (low recall). In this pipeline, Stage 1 is tuned for recall and Stage 3 is tuned for precision.
Geocoding
The process of converting a human-readable place name (such as "Austin, TX") into latitude and longitude coordinates that can be plotted on a map.
Nominatim
The free geocoding service operated by the OpenStreetMap project. Used in Stage 4 of the pipeline. Subject to a one-request-per-second rate limit that the pipeline enforces with a 1.1-second pause between calls.
Folium
A Python library that produces interactive HTML maps from Python objects. Renders entirely on the client; no map server is required at runtime.
